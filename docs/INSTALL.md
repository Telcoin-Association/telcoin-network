# Installing telcoin-network

This guide is for **operators** running a node. It covers:

- Pulling a verified release tarball from GitHub Releases.
- Pulling a verified Docker image from GitHub Packages (`ghcr.io`).
- The exact verification commands that prove the binary you just downloaded
  came from a tagged commit on this repo, signed by both maintainers.

If you're a maintainer cutting a release, see [RELEASING.md](RELEASING.md).

## Trust model

Every release carries two independent attestations:

1. **GitHub-native build provenance.** Generated by `actions/attest-build-provenance`
   in CI. It is bound by OIDC to the workflow file + the commit at the tag.
   Anyone can verify it without a key — `gh attestation verify` checks Sigstore.
2. **Two maintainer countersignatures.** Each maintainer's YubiKey (PIV slot
   9c, ECCP256, touch-required) signs every tarball, the `SHA256SUMS` file,
   and the image manifest digest. A release with only one signature is **not**
   published — the draft sits unmerged.

To trust a release you should pass both verifications. The first proves CI
built it from this repo. The second proves two humans approved it.

## Path A: install from a release tarball

### 1. Pick a release

Browse <https://github.com/telcoin-association/telcoin-network/releases> and
note the tag (e.g. `v0.6.0` for mainnet, `v0.6.0-adiri` for testnet). Set:

```bash
export TAG=v0.6.0
export ARCH=x86_64-unknown-linux-gnu      # or aarch64-unknown-linux-gnu
export ASSET=telcoin-network-${TAG}-${ARCH}.tar.gz
```

### 2. Download

```bash
gh release download "$TAG" \
  --repo telcoin-association/telcoin-network \
  --pattern '*'
```

That fetches the tarball, the matching `.sha256`, `SHA256SUMS`, and every
maintainer signature + cert.

### 3. Verify

```bash
# checksum
sha256sum --check "${ASSET}.sha256"

# 3a. CI build provenance
gh attestation verify \
  --owner telcoin-association \
  --repo telcoin-association/telcoin-network \
  "$ASSET"

# 3b. Maintainer countersignatures (two-of-two)
git clone --depth=1 --branch "$TAG" \
  https://github.com/telcoin-association/telcoin-network tn-verify
for s in grantkee sstanfield; do
  cosign verify-blob \
    --key "tn-verify/.github/release-keys/${s}.pem" \
    --signature "${ASSET}.${s}.sig" \
    --insecure-ignore-tlog \
    "$ASSET"
done
```

`--insecure-ignore-tlog` is correct here: maintainer signatures are
deliberately *not* on Rekor (CI provenance already is, and maintainer
identities are pinned by the certs committed at the tag).

If both maintainer verifications print `Verified OK`, you have a release
that:

- was built by CI from a commit on `telcoin-association/telcoin-network`
- and was approved by both `grantkee` and `sstanfield` after that build

### 4. Extract and run

```bash
tar -xzf "$ASSET"
./telcoin-network --version
```

The version string comes from `VERGEN_GIT_DESCRIBE` baked at build time and
should match `$TAG`.

## Path B: pull a Docker image

```bash
export TAG=v0.6.0
export REPO=ghcr.io/telcoin-association/telcoin-network

# Resolve the manifest digest. Always sign and verify by digest, not by tag —
# tags are mutable, digests are not.
docker buildx imagetools inspect "${REPO}:${TAG}" \
  --format '{{ json .Manifest }}' \
  | jq -r .digest > digest.txt
DIGEST="$(cat digest.txt)"

# Provenance
gh attestation verify \
  --owner telcoin-association \
  --repo telcoin-association/telcoin-network \
  "oci://${REPO}@${DIGEST}"

# Maintainer signatures
git clone --depth=1 --branch "$TAG" \
  https://github.com/telcoin-association/telcoin-network tn-verify
for s in grantkee sstanfield; do
  cosign verify \
    --key "tn-verify/.github/release-keys/${s}.pem" \
    --insecure-ignore-tlog \
    "${REPO}@${DIGEST}"
done

# Pull and run by digest
docker pull "${REPO}@${DIGEST}"
docker run --rm "${REPO}@${DIGEST}" telcoin --version
```

## A shortcut: `make release-verify`

If you cloned the repo at the tag, you can run all of the above with one
command:

```bash
git clone --branch "$TAG" --depth=1 \
  https://github.com/telcoin-association/telcoin-network
cd telcoin-network
make release-verify TAG=$TAG
```

That downloads every asset on the release into a tempdir and runs the same
provenance + cosign checks the maintainers run before publishing.

## Failure modes

| Symptom | What it means |
|---------|---------------|
| `gh attestation verify` fails | The tarball was not built by this repo's release workflow on the commit this tag points at. **Do not run it.** |
| `cosign verify-blob` fails for one signer | The other maintainer hasn't signed yet, or the cert in `.github/release-keys/` was rotated after the release was cut. Check the release page; if both `*.sig` and `*.pem` are attached and verification still fails, treat the release as compromised and report it. |
| `cosign verify` (image) fails | Image digest does not match what either maintainer signed. Don't pull. |
| `--version` doesn't match `$TAG` | The tarball wasn't built from a tagged commit. |

For anything that doesn't add up, file a private security report per
[SECURITY.md](../SECURITY.md). Do not run an unverified binary.

## What about the existing GCP image?

`us-docker.pkg.dev/telcoin-network/tn-public/adiri:latest` is the legacy push
target. It still exists and `make docker-adiri` still pushes to it; the
maintainers use it for internal infrastructure. It is **not** a public release
surface — there's no signature on it, no provenance, and no guarantee it
matches a tag. For verifiable releases, use the GitHub Releases path or
`ghcr.io` as documented above.
