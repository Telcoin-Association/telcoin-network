# Telcoin Network Whitepaper — Topic Outline

- abstract
  - PoS blockchain composed of three ideas
    - DAG-based BFT consensus
    - deterministic EVM execution
    - Narwhal mempool + Bullshark consensus over extended Reth
  - high throughput via decoupling dissemination from ordering
    - horizontal scaling with added worker nodes
  - determinism thesis
    - same consensus output yields identical blocks across validators
- introduction
  - the problem
    - leader-based blockchains
      - single proposer selects and orders txs
      - throughput coupled to one node per slot
    - redundant network dissemination
      - same tx data re-sent during ordering
      - network pays per-tx bandwidth repeatedly
    - underutilized non-leader nodes
    - root cause: dissemination coupled to ordering
      - horizontal scaling impossible
  - the solution
    - separate concerns
      - workers — disseminate batches, prove availability (2f+1)
      - primaries — DAG-BFT ordering over batch digests
      - bullshark — deterministic linearization into sub-DAGs
      - engine — deterministic EVM block materialization (Reth)
      - epochs — on-chain reward payout + committee rotation
    - scale horizontally by adding workers
  - contributions
    - deterministic DAG-to-EVM materialization
    - protocol-executed on-chain epoch governance
    - committee-gated networking + Byzantine penalty model
    - epoch-scoped storage rotation
- system model & protocol overview
  - actors and roles
    - validators — full consensus participants (one primary + workers)
    - primary — DAG-BFT node, headers/votes/certs, Bullshark
    - workers — collect, batch, disseminate, prove availability
    - clients — submit txs to workers
    - observers / full nodes — follow + execute, no consensus
  - node anatomy
    - consensus subsystem (primary + workers)
    - execution subsystem (Reth EVM)
    - networking subsystem (libp2p gossip + req-res)
    - storage subsystem (consensus + execution DBs)
  - fault and timing model
    - partially synchronous BFT
    - f Byzantine of n = 3f+1
    - quorum = 2f+1 stake-weighted votes
    - partial synchrony — eventual delivery, unknown finite bound
    - all thresholds stake-weighted
  - keystone data flow
    - client → worker tx submission
    - worker batches + disseminates to peers
    - peers ack → proof of availability
    - primary embeds batch digest in header
    - header broadcast → votes → 2f+1 certificate
    - certificates form DAG → Bullshark orders
    - ordered output → EVM blocks → execution + state update
  - notation and conventions
    - n total validators, f Byzantine faults
    - 2f+1 quorum (supermajority by stake)
    - f+1 weak quorum (≥1 honest)
    - H(·) cryptographic digest
- consensus layer
  - narwhal data availability
    - availability proven before ordering
    - workers and batches
      - txs grouped into digest-identified batches
      - broadcast to peer worker on every validator
      - await 2f+1 stake-weighted acks
      - acks form proof of availability
    - batch dissemination and availability
      - batches persisted locally, served on request
      - retrievable for execution after DAG inclusion
  - primary — headers, votes, certificates
    - header construction
      - primary id + round number
      - batch digests from own workers
      - prior-round certificates (parents)
      - timestamp + metadata
    - header voting
      - validate header → signed vote
      - collect until 2f+1 stake quorum
    - certificate formation
      - 2f+1 votes → certificate
      - certificates are DAG vertices
  - proposer and rounds
    - one header per primary per round
    - round advancement
      - advance to r+1 on 2f+1 certs from round r
      - those certs become next-round parents
    - garbage collection
      - drop old rounds → bounded DAG memory
  - bullshark ordering
    - anchors and commit rule
      - periodic anchor vertex selection
      - commit on f+1 supporting certs from next round
      - causal history ordered before anchor
    - sub-DAG linearization
      - deterministic causal-history traversal
      - yields linear certificate sequence (sub-DAG)
  - safety and liveness
    - safety — no conflicting commits (quorum intersection)
    - liveness — continued commits under partial synchrony
  - TN-specific consensus deltas
    - stake-weighted quorums (not node count)
    - epoch-aware fixed committees
    - dedicated consensus output channel to execution
- execution layer
  - consensus-to-execution boundary
    - consensus emits ordered certs (sub-DAGs)
    - execution consumes → EVM blocks
    - consensus output contents
      - anchor cert + causal history
      - referenced batches
      - processing order
  - TNPayload translation
    - all inputs for deterministic block build
    - deterministic block construction
      - identical block from identical consensus output
      - all inputs derived from consensus output
    - repurposed header fields
      - mix_hash — consensus output digest
      - nonce — worker/batch index
      - difficulty — consensus round number
      - extra_data — additional consensus metadata
      - timestamp — consensus-derived, not wall-clock
  - base fee and gas model
    - deterministic base fee
    - independent of local state, derived from consensus output
  - Reth integration
    - reused components
      - EVM executor (revm)
      - state + storage management
      - transaction pool (adapted)
      - RPC server
    - replaced components
      - block production → TNPayload translation
      - consensus → DAG-BFT
      - fork choice → deterministic consensus output
  - design trade-offs
    - determinism over flexibility → independent verification
    - header field repurposing → Ethereum tooling compatibility
- epochs & validator lifecycle
  - what is an epoch
    - fixed period of constant committee
    - boundary triggers committee change, rewards, stake updates
  - epoch transition
    - atomic transition
      - all validators agree on exact boundary block
      - determined from consensus output
    - epoch boundary block
      - final block triggers transition
      - carries closing + setup system calls
  - system contracts
    - ConsensusRegistry — validator set, stakes, membership
    - Issuance — reward issuance + distribution
  - epoch system calls
    - system call mechanism
      - special protocol-executed txs (not user-submitted)
      - system caller address, elevated privileges
    - epoch closing calls
      - calculate + distribute rewards
      - update validator stakes
      - determine next committee
  - committee selection and shuffling
    - deterministic shuffle
      - committee from stakes + consensus-derived shuffle
      - all validators compute identical committee
  - token economics and rewards
    - reward calculation
      - function of stake + participation
      - formula in Issuance contract
  - validator lifecycle state machine
    - pending — registered, not active
    - active — committee member in consensus
    - exiting — winding down
    - exited — fully left
  - node restart and recovery
    - recovery process
      - load persisted consensus + execution state
      - determine current epoch + round
      - re-sync with peers to catch up
  - design trade-offs
    - on-chain governance → transparency vs system-call care
    - epoch-based committees → simplicity vs slow set change
- networking layer
  - network stack and topology
    - libp2p
    - committee-organized connectivity
  - gossipsub and authorized publishers
    - gossipsub broadcast
    - publisher authorization
      - per-topic authorized publisher set
      - reject unauthorized → block injected consensus messages
  - request-response RPC
    - RPC protocols
      - fetch batches by digest
      - fetch certificates
      - state synchronization
  - kademlia and peer discovery
    - Kademlia DHT, validators publish addresses
    - peer RPC info records
      - published to DHT for RPC endpoint discovery
  - bulk streaming
    - dedicated protocols for large transfers (state sync)
  - peer scoring and penalties
    - behavior-based scoring
    - penalty model
      - scores from behavior
      - persistent misbehavior → disconnect + ban
  - state synchronization
    - fetch missing certificates + batches from peers
  - design trade-offs
    - committee-gated networking → security vs membership management
    - libp2p → robustness vs complexity
- storage layer
  - dual database architecture
    - consensus DB — headers, certificates, votes
    - execution DB — accounts, storage, blocks
  - table taxonomy
    - Headers
    - Certificates
    - Votes
    - Batches
    - LastProposed
  - layered write path
    - in-memory buffering → DB persistence
  - archival and crash safety
    - persist critical state before acting
    - recover from persisted state on restart
  - design trade-offs
    - dual databases → independent optimization vs coordination
- security & correctness properties
  - Byzantine fault tolerance (f of 3f+1)
  - safety — no conflicting committed state
  - liveness — progress under partial synchrony
  - determinism — identical state from same consensus output
  - data availability — proven before ordering
  - censorship resistance — DAG structure
- performance & scalability
  - horizontal scaling via added workers
  - decoupled dissemination removes leader bottleneck
  - parallel batch processing
  - pipelined consensus over the DAG
- related work & comparison

  - Bullshark (consensus ordering)
  - DAG-Rider (earlier DAG-BFT)
  - traditional BFT — PBFT, Tendermint (leader-based)
  - Ethereum (execution model via Reth)
- conclusion & future work
  - DAG-BFT + deterministic EVM → horizontally scalable chain
  - future work

    - validator set expansion
    - networking enhancements
