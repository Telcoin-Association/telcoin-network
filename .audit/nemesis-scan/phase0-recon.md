## Attack Goals (Q0.1)

A connected peer in the libp2p mesh that has cleared the network handshake but is byzantine at the application layer can plausibly aim for:

1. **Persistent consensus-DB index pollution / "orphan-batch scar".** `Inner::stream_import` (`crates/storage/src/consensus_pack.rs:737-749`) writes a `Batch` record AND inserts it into the `batch_digests` HdxIndex BEFORE the next `Consensus` record validates the parent-hash chain. On any post-batch error path (CRC mismatch, RST, malformed next record, timeout at `consensus_pack.rs:684-689`) the bytes already landed in the data file and the index. Although `consensus.rs:432-437` does `remove_dir_all(&path)` on error, the import-staging dir is `import-{epoch}` (`consensus.rs:377`) — well-isolated, but only if the `Err` branch is hit. A panic inside the `while let Some(record) =` loop (`consensus_pack.rs:731`) leaves the temp dir behind and will be re-created on the next attempt only if `remove_dir_all` succeeds at the start of the next `stream_import` call. Cross-epoch confusion is also reachable if the staging dir contains data from a previous failed peer.

2. **Unbounded heap allocation → process OOM (CRITICAL).** `AsyncPackIter::read_record_file` (`crates/storage/src/archive/pack_iter.rs:148-150`) reads a peer-supplied 4-byte little-endian `val_size: u32`, then calls `buffer.resize(val_size as usize, 0)` with **no length cap and no comparison against any sane batch/header upper bound**. A malicious peer can claim `val_size = 0xFFFF_FFFF` (~4 GiB) per record. The per-record tokio timeout (`PACK_RECORD_TIMEOUT_SECS = 10` from `state-sync/src/consensus.rs:14`) wraps the whole `read_record_file` future, but allocator behavior on an unsuccessful 4 GiB request varies across OSes (Linux overcommit + OOM-killer vs. macOS hard ENOMEM). On constrained nodes this is straight-line crash; on overcommit Linux it allows the attacker to walk the node into swap and missed liveness deadlines. Same pattern in the sync iterator at `pack_iter.rs:75-77` and in `Pack::read_record` (`pack.rs:319-321`) — but the network path is `AsyncPackIter`.

3. **Concurrency-slot exhaustion / honest-peer starvation.** With `MAX_CONCURRENT_EPOCH_STREAMS = 5` and `MAX_PENDING_REQUESTS_PER_PEER = 2` (`primary/src/network/mod.rs:60,65`), three colluding malicious peers each holding 2 slots = 6 > 5 → all global semaphore permits held. Each slow-streamer feeds the minimum legal pace: one record every <10s avoids the per-record timeout (`consensus_pack.rs:684`) but there is **no whole-stream cap**. The 30s `PENDING_REQUEST_TIMEOUT` (`network/mod.rs:54`) only governs the *pre-stream-open* wait, after which the entry is removed by `process_inbound_stream` (`network/mod.rs:798-800`); once the stream is attached, the prune loop's `retain` (`mod.rs:493-495`) checks `created_at` but the entry has already been removed at stream-open. Honest peers see `try_acquire_owned()` fail → `ack=false` → exponential give-up. Combined with the predictable `request_digest` (see #5), this is a deterministic DoS.

4. **`request_digest` is non-binding & predictable → cross-session/cross-peer slot hijack.** `request_digest = digest(epoch.to_le_bytes())` only (`network/mod.rs:334-336` and `network/mod.rs:690-692`). It contains **no committee id, no requester BLS pubkey, no nonce, no protocol version**. Any peer that knows the epoch number — which is public — can compute the same digest. Combined with the `(BlsPublicKey, B256)` pending-map key (`mod.rs:81`), the BLS in the key is the *responder*'s claimed pubkey, not the libp2p peer-id. The key is `(peer, request_digest)` where `peer` IS the libp2p PeerId-derived BLS, but a peer that opens an inbound stream and writes any 32 bytes of a known epoch digest can match a legitimate pending entry IF that peer is the same libp2p peer the requestor sent the original request to (which the attacker can be, since they're the responder). The risk is more subtle: if a single peer is selected for the stream by `send_request_any`, and a subsequent unrelated request to the same epoch produces the same digest, the second attaches the wrong pending entry. Also: digest reuse across reconnects — a peer that lost mid-stream can reconnect and present the same digest to attach to a stale entry that has not yet been pruned.

5. **Penalty mis-attribution / peer-score griefing.** `request_epoch_pack` applies `Penalty::Mild` (`mod.rs:381`) on stream failure to the BLS-claimed `peer` returned in the `RequestEpochStream { ack, peer }` field. If the responder lies — claiming `peer = some_honest_validator` while actually being a different libp2p PeerId on the wire — the penalty hits the honest BLS, not the attacker. `report_penalty(peer: BlsPublicKey, ...)` accepts the BLS as canonical. Also: `Penalty::Mild` is the entire ladder — there is no escalation when the same peer fails repeatedly. The `unknown_stream_request` path (`error/network.rs:117`) is also `Penalty::Mild`. An attacker who consistently spams unmatched inbound streams at +Mild/event indefinitely never escalates.

6. **Watch-channel poisoning during failed import (mid-stream observable poisoned state).** Per `domain-patterns.md` Example 5 — `last_consensus_header` watch is on `ConsensusBusApp` (`consensus_bus.rs:216`). In the streaming-import path I can NOT find any `last_consensus_header().send_replace(..)` inside `stream_import`, so direct mid-stream poisoning of THAT watch is not in this module — BUT `state-sync/src/consensus.rs:64-65` calls `consensus_bus.last_consensus_header().send_replace(Some(header))` on the regular per-header `request_consensus` path. Risk class to validate in Phase 2: does `latest_consensus.update(epoch, number)` (`storage/src/consensus.rs:485`) or any `LatestConsensus.update` get called BEFORE the rename in `stream_import` (`consensus.rs:422`) finishes? Pre-rename, the data sits in `import-{epoch}` and `current_pack` is untouched, so reads should see old state. Phase 2 must verify there is no observable window where `recent_packs` is invalidated before the rename completes (currently `consensus.rs:428` invalidates AFTER rename, which is correct).

## Novel Code (Q0.2)

- **`pack_iter.rs:135-160`** — Hand-rolled `(u32 val_size, val_bytes, u32 crc32)` framing on top of an async `AsyncRead`. No `length_delimited_codec`, no `tokio_util::codec`. Trust boundary entirely on `val_size as usize`.
- **`pack_iter.rs:149-150` and `pack.rs:320-321`** — Resize-then-`read_exact` pattern with no length cap. Same-shape novel deserialization, repeated.
- **`consensus_pack.rs:680-690`** — Bespoke `next()` helper that wraps `tokio::time::timeout` around per-record stream consumption; per-record timeout with no whole-stream timeout.
- **`consensus_pack.rs:729-783`** — Hand-rolled streaming state machine that maintains a `HashSet<batch_digest>` (`batches`, line 730), pops entries as the next consensus header references them (`batches.remove(digest)` line 756), enforces empty-set on each consensus record (line 761). Custom epoch-pack invariant validation, not a library.
- **`network/mod.rs:333-336` and `mod.rs:690-692`** — Custom request-digest derivation: `digest(epoch.to_le_bytes())`. No domain-separation tag, no committee binding, no version.
- **`network/mod.rs:678-721`** — Custom acquire-permit-then-check-and-insert state machine for slot accounting. Permit acquired before per-peer count check, dropped on rejection. The `created_at` preservation logic (`mod.rs:698-701`) is novel: if same `(peer, request_digest)` entry exists, keep old timestamp — a hand-rolled idempotency primitive.
- **`network/mod.rs:466-487`** — Hand-rolled prune interval loop in `tokio::select!` next to event recv. No panic guard, no heartbeat — if the spawned future panics, the entire `process_network_event` task dies (it's spawned as `spawn_critical_task`, which surfaces panics, but the per-request spawned tasks at `task_spawner.spawn_task(...)` (e.g. `mod.rs:778`) are unguarded — a panic in `process_inbound_stream`'s spawned task does NOT remove the entry from `pending_epoch_requests` since it was already removed at line 800 before the await; but it DOES leak the underlying I/O resources and skips the penalty report).
- **`consensus.rs:357-438` (`ConsensusChain::stream_import`)** — Custom rename-on-success workflow: write to `import-{epoch}`, validate pack tail equals `epoch_record.final_consensus`, then atomic-ish `fs::rename` from `path_base_dir` to `base_dir`. The window between `remove_dir_all(&base_dir)` (`consensus.rs:420`) and `fs::rename` (`consensus.rs:422`) is acknowledged in code comments as a hazard; novel atomicity-coupling.
- **`consensus_pack.rs:440-527` (`files_consistent` and `trunc_and_heal`)** — Bespoke recovery logic that reconciles three storage components (`data`, `consensus_idx`, `consensus_digests`/`batch_digests`) when their lengths disagree. Novel torn-write recovery; comments at `consensus_pack.rs:489-493` admit the digest indexes are intentionally left with potentially-orphan entries.
- **`consensus.rs:60-273`** — Hand-rolled dual-slot atomic write of `(Epoch, u64)` to two slot files with CRC; chosen slot determined by reading both at startup and picking max `(epoch, number)`. Replaces what could be a single REDB transaction with bespoke A/B file writes.

## Complex Paths (Q0.4)

### Path 1 — Peer bytes → consensus DB write (the primary attack surface, ~9 hops)

**Length: 9 hops**

`libp2p NetworkEvent::InboundStream` → `PrimaryNetwork::process_inbound_stream` (`primary/src/network/mod.rs:772`) → spawned task reads 32-byte digest (`mod.rs:781-794`) → `pending_map.lock().remove(...)` (`mod.rs:798-800`) → `RequestHandler::process_request_epoch_stream` (`handler.rs:941`) — wait, this is the SERVER path. Let me redo for the CLIENT (the receiving import path):

**Client/receiver path (the actually-dangerous path — bytes flow into our DB):**

1. Application schedules epoch fetch: `state_sync::consensus::spawn_fetch_consensus` reads from `epoch_request_queue` (`state-sync/consensus.rs:178`)
2. → `PrimaryNetworkHandle::request_epoch_pack` (`primary/src/network/mod.rs:322`)
3. → libp2p `send_request_any` (`mod.rs:343`) for `RequestEpochStream` ack
4. → `handle.open_stream(peer)` (`mod.rs:358`) + write 32-byte digest correlator
5. → `ConsensusChain::stream_import` (`storage/src/consensus.rs:357`) — wraps stream, sets up `import-{epoch}` staging dir
6. → `ConsensusPack::stream_import` (`storage/src/consensus_pack.rs:254`) — spawns OS thread for the pack loop
7. → `Inner::stream_import` (`consensus_pack.rs:673`) — opens 4 storage components
8. → `AsyncPackIter::open` reads & validates `DataHeader` (`pack.rs:419-426`); per-record loop calls `next()` (`consensus_pack.rs:680-690`) which calls `iter.next()` → `read_record_file` (`pack_iter.rs:135-160`) which **`buffer.resize(val_size as usize, 0)`** (line 149) and `read_exact` (line 150) directly from peer bytes
9. → For each `Batch`: `data.append(&PackRecord::Batch(batch))` writes to local data file → `batch_digests.save(...)` writes to local index → mutates `data_file_length` markers (`consensus_pack.rs:741-748`). For each `Consensus` header: parent-chain validation, `data.append(...)`, two index inserts (`batch_digests` would be unchanged here, but `consensus_idx` and `consensus_digests` both get a write). Note: NO per-record DB transaction — these are 3 independent file writes.
10. → On full success: `pack.persist().await` (`consensus.rs:386`); validate `final_consensus` matches → `fs::rename(import-N/epoch-N, base/epoch-N)` (`consensus.rs:422`); `recent_packs.lock().retain(...)` invalidate (`consensus.rs:428`)

**Why high-value:** every byte from the peer flows into the data file and three indexes BEFORE any cryptographic validation (only structural validation: parent-hash chain via local `parent_digest` and `batches` set membership). The trust assumption is "if the structure matches the local epoch_record, the bytes are authentic" — but signed validity of the consensus headers themselves is NOT verified inside `stream_import`. Only the FINAL header is checked at `consensus.rs:387-401` (`epoch_record.final_consensus.number == last_header.number && hash == digest()`). Headers in the middle could be fabricated as long as parent-chain links and batch-set bookkeeping balance.

### Path 2 — Peer bytes → watch-channel signal → downstream subscribers (~6 hops, mediated by gossip)

`libp2p Gossip(PrimaryGossip::Consensus)` → `PrimaryNetwork::process_gossip` (`mod.rs:752`) → `RequestHandler::process_gossip` (`handler.rs:193`) → `consensus_certs.insert/get` quorum tally (`handler.rs:265-289`) → upon ≥1/3+1 sigs: `last_published_consensus_num_hash().send_replace(...)` (`handler.rs:281-282`) → subscriber `state-sync::consensus::spawn_track_recent_consensus` `rx_gossip_update.changed()` (`state-sync/consensus.rs:102`) → `consensus_bus.request_epoch_pack_file(epoch_record).await` (`consensus.rs:116`) → enters `epoch_request_queue` → Path 1.

**Length: 6 hops.** Why high-value: this is the *trigger* for Path 1. A malicious peer that can spoof or rush the gossip flow can drive the receiver's epoch-pack queue, in combination with attacker-controlled stream responses, to maximize the OOM/scar window.

### Path 3 — Peer bytes → `report_penalty` → score state (5 hops)

Inbound stream → `process_inbound_stream` task → fail in `process_request_epoch_stream` (`handler.rs:941`) → return `PrimaryNetworkError::UnknownStreamRequest` (`handler.rs:958`) → `(&err).into()` ⇒ `Penalty::Mild` (`error/network.rs:117`) → `network_handle.report_penalty(peer, ...)` (`mod.rs:809`).

This is shorter but penalty-state coupling is the griefing vector (#5 in Attack Goals).

## Attacker's Hit List (priority for Phase 2)

Ranked by `(impact × novelty × complexity)`:

1. **`pack_iter.rs:148-150` (`AsyncPackIter::read_record_file`) — UNCAPPED `buffer.resize(val_size as usize, 0)`.** Impact: process kill / OOM / persistent DoS. Novelty: hand-rolled framing with no length cap. Complexity: trivial to exploit (one 4-byte write). This is the lowest-hanging exploit in the entire branch. **First Feynman target.**

2. **`consensus_pack.rs:737-783` (`Inner::stream_import` per-record write loop) — multi-table writes outside any transaction.** Three writes per consensus record (`data.append`, `consensus_digests.save`, `consensus_idx.save`) and two per batch (`data.append`, `batch_digests.save`) — none atomic. Recovery relies on `trunc_and_heal` which intentionally leaves orphan digest entries (`consensus_pack.rs:489-493`). Combined with the `Err` path at `consensus.rs:432-437`'s `remove_dir_all` clean-up, evaluate: **does an unwind/panic between the data.append and the index.save leave the data file longer than `data_file_length` markers, and is `trunc_and_heal` correctly triggered the next time `open_append` runs on this directory?** Note: in `stream_import` the directory is the staging `import-{epoch}` and is removed on Err — but does the normal `Drop` of `PackInner` auto-`commit` (`pack.rs:163-166`) BEFORE the staging dir is removed, leaking the staged batch into the data file's fsync? **Critical state-inconsistency target.**

3. **`network/mod.rs:334-336, 690-692` — `request_digest = digest(epoch.to_le_bytes())` lacks domain separation.** Impact: cross-peer / cross-session correlation collisions, replay across reconnects. Novelty: custom digest with insufficient binding. Combined with `Penalty::Mild` for `UnknownStreamRequest`, an attacker can probe pending-map state at low cost. **Phase 2: enumerate every state transition where a stale or attacker-known digest can attach to a legitimate slot.**

4. **`network/mod.rs:798-812` (`process_inbound_stream`) — permit released BEFORE import starts.** When the entry is removed from `pending_map` at line 800, the `PendingEpochStream` is dropped, releasing its `_permit`. The actual `process_request_epoch_stream` runs AFTER this point. Hypothesised invariant `permits + active_imports == MAX_CONCURRENT_EPOCH_STREAMS` is violated — once a peer attaches its stream, the slot is freed for another acceptance, and N peers can have N active-but-unaccounted imports simultaneously. **Concurrency invariant target.**

5. **`consensus.rs:417-422` (`stream_import` rename window)** — `if std::fs::exists(&base_dir) { remove_dir_all(&base_dir); } std::fs::rename(&path_base_dir, &base_dir);`. The acknowledged TOCTOU window between `remove_dir_all` and `rename`: any concurrent reader (e.g. `get_static` scanning `recent_packs`) opening the old `epoch-{N}/data` inode just before unlink will hold an FD on a deleted file; new readers between unlink and rename get `ENOENT`. The `recent_packs` invalidation at line 428 happens AFTER rename — by design — but a reader between lines 420 and 422 could observe a missing pack and a separate reader between 422 and 428 could read NEW pack but see the OLD entry in `recent_packs` cache. **Atomicity-coupling target.**

6. **`pack_iter.rs:148-150` — `val_size as usize` truncation on 32-bit builds.** Secondary concern: on a 32-bit target, `u32 as usize` is identity, so the cap is u32::MAX (~4 GiB) which is allocator-realistic. On 64-bit no truncation but no upper bound either. Either way, **no comparison against any constant**.

7. **`error/network.rs:115-118` — `Penalty::Mild` for both `UnknownStreamRequest` and `InvalidRequest`.** Combined with no penalty escalation table, an attacker can sustain misbehavior indefinitely. **Penalty-arithmetic target — integrate with #3 to estimate attacker cost.**

8. **`network/mod.rs:341-345` — `peer` field returned by `RequestEpochStream` response is application-layer claim, not transport-bound BLS.** If `report_penalty` (`mod.rs:381`) penalizes that claimed BLS, mis-attribution. **Identity-binding target.**

9. **`consensus_pack.rs:730-783` — `batches: HashSet<BlockHash>` is monotonically populated and only emptied per consensus header.** If the malicious peer sends `(batch1, batch2, ..., batchN, header_for_batch1)` (header references only batch1), the loop fails at `batches.is_empty()` check (`consensus_pack.rs:761`) — good — but only AFTER batch2..batchN are already in `data` and `batch_digests`. Phase 1 may show that `import-{epoch}` cleanup recovers this; Phase 2 must confirm no path lets the staging dir be promoted partial. Also: between failure and `remove_dir_all`, an OS crash leaves orphan files. **Worked-example #1 from domain-patterns realized.**

10. **`mod.rs:797-800` — `pending_map.lock().remove(...)` BEFORE the spawned task verifies the stream is well-formed.** A flood of unmatched inbound streams produces N `Penalty::Mild` reports without invoking the importer, rapidly spending the misbehavior budget. With predictable `request_digest` from #3, attacker can also craft inbound streams that DO match a legitimate pending entry — yielding the slot to the attacker mid-flight. **Slot-hijack target.**
